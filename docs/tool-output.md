# Tool Output

Read this when writing or modifying any tool's `execute()` return value, error handling, or partial-success behavior.

A tool returns `{ content, details }`. Treat the two channels as distinct audiences.

- **`content` → the model.** Pi catches anything `execute` throws and tags the result `isError: true`; providers map that to native error semantics (Anthropic `is_error: true`, Bedrock `ToolResultStatus.ERROR`, Google `{ error }`). Returning a string with a "this is an error" sentence does NOT - the model receives it as a normal success and cannot structurally distinguish it from real tool output. **Throw for every error and edge case** (not found, empty, out-of-range, permission denied, etc.); never return an English-prose error string in `content`.
- **`details` → renderers and programmatic consumers, not the model.** Providers don't serialize `details` over the wire. Put pagination metadata, byte counts, file paths, structured truncation info, diff objects (`ToolDiff`), exit codes, etc. here so the TUI/renderer can display affordances without that text leaking into `content`. See how `bash` stores exit/signal info in `details` via `detailsOf()` (`src/extensions/bash/format.ts`) and how `edit`/`write` carry the diff object so `DiffView.renderDiffResult` can syntax-highlight it.
- **Partial-success affordances**: the model needs the hint, so it has to live in `content`. Use square brackets `[…]` as the clearly-fenced bracket form — e.g. `[read tool: showing lines 1-50 of 200; call read again with start=51 to continue.]`. When the tool returns a body + truncation footer, emit them as separate `content` blocks (`[{type:'text', text: body}, {type:'text', text: '[...]'}]`) — the model sees both (Anthropic concatenates them), and renderers can detect the two-block pattern for UI polish. Mirror the structured fields into `details`. This pattern is used by `read`, `glob`, `grep`, and `web_fetch`. When the full output is too large to keep inline, spill it to disk via `SpillCache` (`src/shared/SpillCache.ts`) and point the footer at the saved path with a resume line (`use read with start=N …`, computed from `Lines.continuationLine` over the inline head so it lands on `read`'s line numbering) instead of asking the model to re-run/re-fetch; `bash` and `web_fetch` both do this. `SpillCache` owns the `~/.pim/cache` dir, 0600 writes, and the TTL sweep.
- **Error messages should be actionable.** Include the underlying error code, name what failed, and where useful suggest the next tool call (e.g. "use glob to locate the file"). Canonical patterns: `read.ts` (range/type errors + file-not-found via `FsErrors`), `edit.ts` (not-found with closest-region candidates, overlap detection, all-noop), `bash/run.ts` (exit code + signal handling).
- **File-not-found errors go through `FsErrors.statOrThrow`** (`src/shared/FsErrors.ts`). It throws a clean `Path not found: …` message with up-to-three "did you mean" sibling suggestions from the parent directory. Use it for any file/dir stat that the model triggers.
- **Argument-validation errors are handled by `Tools.wrap`/`register`** (see [pi-api.md](./pi-api.md)). Your `execute` only sees args that passed schema validation, so don't repeat type checks there. Throw for _semantic_ errors (file not found, range out of bounds, conflict, etc.); throw for input _type_ errors only if your schema can't express the constraint.
- **`cwd` param validation**: `bash`, `grep`, and `glob` accept a `cwd` param that must be an absolute path. Validate with `Paths.requireAbsolute(cwd)` — it throws `Path must be absolute, not relative: …` for relative paths. The render title shows ` (in: <relative>)` when cwd differs from the workspace root (via `Paths.cwdSuffix`). See `src/extensions/bash/index.ts`, `src/extensions/grep/index.ts`, `src/extensions/glob/index.ts`.
- **Clickable file links in titles**: `renderFileLink` (in `Renderer`) emits OSC 8 `file://` hyperlinks so terminal Alt+Click opens the file. All read/edit/write/apply-patch titles pass `clickableLink: true` (or default). When adding a new tool that shows a file path in its title, use `Renderer.renderFileLink(theme, display, absolute)` (clickable defaults to `true`) or `DiffView.renderDiffCall({ link: true, clickableLink: true })`.
